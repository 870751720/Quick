# 浏览器原型技术说明

状态：生效中。使用 Canvas 与原生 ES Modules。`src/combat.js` 保存可测试规则，`src/game.js` 负责输入、状态推进与渲染。

Node.js 20+ 执行 `npm start`，访问 `http://localhost:8080`。验证使用 `npm test` 和 `npm run check`。当前已有浏览器本地轻量存档与程序音效，但没有地图碰撞、手柄、日志持久化或遥测；出现复杂关卡工具链、动画状态机或多平台发布需求时迁移专业引擎。

人工体验统一通过 GitHub Pages 进行；本地服务仅供 Agent 必要的工程诊断，不作为用户验收入口。`main` 分支推送通过 GitHub Actions 自动执行测试并部署 Pages。

GitHub Pages 默认可能缓存静态文件约 10 分钟。发布修改过的 JS/CSS 时必须同步递增 `index.html` 中的资源版本参数；ES Module 的所有传递依赖也必须使用相同版本参数，不能只更新入口脚本。交付链接附加本次 commit 或版本查询参数，以避免人工体验命中旧缓存。
装备生成、背包容量、穿戴交换与属性汇总集中在 `src/inventory.js`，不依赖 Canvas 表现层，可由 Node 单元测试直接验证。`game.js` 只负责掉落接入、交互和绘制。
装备估价、单件出售和批量出售同样由 `src/inventory.js` 提供纯函数；商人距离判定、暂停、面板点击和反馈由 `game.js` 负责。批量出售只遍历背包格，不读取装备栏，从结构上避免误售已穿戴装备。

背包事务层负责堆叠、溢出占格、任意数量拆分、换位/合并、消耗、排序、筛选、锁定和废品出售。堆叠兼容条件由 `definitionId` 决定；拆分生成新实例 ID。表现层只能调用这些接口，不直接实现数量运算。当前为单机内存状态；未来存档或联网时应在事务边界增加版本与服务端校验。
# GM 调试边界

GM 功能位于 `src/gm.js`，采用命令注册表隔离 Canvas UI 与游戏状态调试接口。公开 Pages 默认不启用，使用 `?gm=1` 显式启用并按 F1 打开；这是开发验收入口而非权限控制。面板操作暂停游戏，命令结果和错误显示于面板并保留最近 50 条会话日志，不写入持久存储。
