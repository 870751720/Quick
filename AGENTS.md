# AGENTS.md

本文件定义 AI Agent 在本项目中的工作方式、协作原则与执行边界。除非用户明确提出例外，所有 Agent 都必须遵守本文件。

## 1. 沟通对象与协作原则

- 用户是一名拥有 6 年工作经验的游戏程序员，熟悉游戏客户端、服务端及 Web 开发，具备扎实的工程基础。
- 默认以资深工程师之间的方式沟通。无需解释常识性概念；涉及复杂设计、隐含风险或关键取舍时，应清晰说明依据。
- 不盲从用户提出的方案。发现需求、设计、实现路径或技术判断存在不合理之处时，必须直接指出，并说明原因、影响及可执行的替代方案。
- 可以主动质疑假设、暴露风险和提出不同意见，不必担心冒犯用户；但应基于事实和工程判断，避免无依据的争论。
- 用户结果导向。所有讨论与决策都应服务于项目目标，优先选择能够可靠完成目标的方案，而不是刻意迎合、炫技或拘泥于既定思路。
- 当多个方案都可行时，应明确推荐其中一个，并简要说明关键权衡；不要只罗列选项而回避判断。
- 讨论出现分歧时，以可验证的证据为准，包括代码、测试、性能数据、官方文档、原型结果和实际运行表现。

## 2. Git 提交与 CI/CD

- 每个完整的修改任务都必须形成一个可验证的交付单元。完成实现和必要验证后，必须创建 Git commit，不得仅将改动留在工作区。
- 提交前必须检查改动范围和验证结果，避免把无关文件、临时文件、生成缓存、密钥或本地环境配置混入提交。
- Commit message 应准确描述本次交付内容；一个提交只承载一个逻辑目标。若任务包含多个相互独立的目标，应拆分提交。
- 项目必须基于 GitHub 建立 CI/CD 流程。技术栈与部署目标确定后，应尽早通过 GitHub Actions 实现自动构建、自动测试以及适用于目标平台的发布或部署流程。
- CI/CD 配置必须与本地构建方式保持一致，关键步骤应可在本地复现。不得通过跳过测试、忽略错误或硬编码敏感信息来制造表面成功。
- 每次推送触发 CI/CD 后，Agent 应跟踪运行状态直至结束，并向用户同步成功或失败结果；失败时应同时报告失败阶段、核心原因和下一步处理方案。
- 若因缺少 GitHub 远端、访问权限、Secrets、签名证书、发布账号或其他外部条件而无法提交、推送或完成 CI/CD，必须明确说明阻塞项，不得声称流程已经完成。
- “修改完成”默认指实现、验证、提交以及已配置流水线下的 CI/CD 检查均已完成。若当前阶段尚未具备 CI/CD 条件，必须在交付说明中明确标注。

### 2.1 GitHub 凭据位置与安全要求

- GitHub 凭据统一通过 GitHub CLI（`gh auth login`）保存到操作系统凭据库；Agent 通过 `gh` 使用凭据，不在项目文件中保存或读取明文 Token。
- 若 GitHub CLI 暂不可用，本项目约定本地 GitHub Token 文件位于 `D:\Quick\token`。该文件仅用于本机鉴权，已由根目录 `.gitignore` 忽略；不得提交、复制到其他项目文件或在输出中展示其内容。
- `D:\Quick\token` 中只能存放未泄露且仍有效的 Token。Agent 使用前应避免回显，并在发现凭据已暴露、失效或权限过宽时停止使用并要求轮换。
- `AGENTS.md`、源码、配置文件、提交记录、构建日志和对话中均不得记录真实 Token、密码、私钥或其他敏感值。
- GitHub Actions 运行时所需的第三方凭据必须保存为 Repository、Environment 或 Organization Secrets，并在工作流中通过 `secrets` 上下文引用。
- 本地工具确需环境变量时，仅记录变量名和配置方法，不记录变量值；优先使用当前进程级变量或操作系统安全存储，禁止提交包含真实值的 `.env` 文件。
- 任何曾出现在对话、日志、截图或版本库中的凭据都必须视为已泄露，立即撤销并重新生成，不得继续使用。
