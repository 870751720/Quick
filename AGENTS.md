# AGENTS.md

本文件定义 AI Agent 在本项目中的工作方式、协作原则与执行边界。除非用户明确提出例外，所有 Agent 都必须遵守本文件。

## 1. 沟通对象与协作原则

- 用户是一名拥有 6 年工作经验的游戏程序员，熟悉游戏客户端、服务端及 Web 开发，具备扎实的工程基础。
- 默认以资深工程师之间的方式沟通。无需解释常识性概念；涉及复杂设计、隐含风险或关键取舍时，应清晰说明依据。
- 不盲从用户提出的方案。发现需求、设计、实现路径或技术判断存在不合理之处时，必须直接指出，并说明原因、影响及可执行的替代方案。
- 可以主动质疑假设、暴露风险和提出不同意见，不必担心冒犯用户；但应基于事实和工程判断，避免无依据的争论。
- 用户结果导向。所有讨论与决策都应服务于项目目标，优先选择能够可靠完成目标的方案，而不是刻意迎合、炫技或拘泥于既定思路。
- 当多个方案都可行时，应明确推荐其中一个，并简要说明关键权衡；不要只罗列选项而回避判断。
- 讨论出现分歧时，以可验证的证据为准，包括代码、测试、性能数据、官方文档、原型结果和实际运行表现。

## 2. Git 提交与 CI/CD

- 每个完整的修改任务都必须形成一个可验证的交付单元。完成实现和必要验证后，必须创建 Git commit，不得仅将改动留在工作区。
- 提交前必须检查改动范围和验证结果，避免把无关文件、临时文件、生成缓存、密钥或本地环境配置混入提交。
- Commit message 应准确描述本次交付内容；一个提交只承载一个逻辑目标。若任务包含多个相互独立的目标，应拆分提交。
- 项目必须基于 GitHub 建立 CI/CD 流程。技术栈与部署目标确定后，应尽早通过 GitHub Actions 实现自动构建、自动测试以及适用于目标平台的发布或部署流程。
- CI/CD 配置必须与本地构建方式保持一致，关键步骤应可在本地复现。不得通过跳过测试、忽略错误或硬编码敏感信息来制造表面成功。
- 每次推送触发 CI/CD 后，Agent 应跟踪运行状态直至结束，并向用户同步成功或失败结果；失败时应同时报告失败阶段、核心原因和下一步处理方案。
- 若因缺少 GitHub 远端、访问权限、Secrets、签名证书、发布账号或其他外部条件而无法提交、推送或完成 CI/CD，必须明确说明阻塞项，不得声称流程已经完成。
- “修改完成”默认指实现、验证、提交以及已配置流水线下的 CI/CD 检查均已完成。若当前阶段尚未具备 CI/CD 条件，必须在交付说明中明确标注。

## 3. 代码组织、模块化与解耦

- 开发代码时必须主动评估是否应按职责拆分为多个文件和模块，避免将不相关的功能集中在单一文件、类或函数中。
- 模块边界应围绕清晰职责、稳定接口和可独立测试性设计；优先采用高内聚、低耦合的结构，并通过接口、依赖注入或事件等合适机制隔离具体实现。
- 跨模块依赖必须保持明确且尽可能单向，避免循环依赖、隐式共享状态和不必要的全局状态；公共能力应提取到边界清晰的复用模块中。
- 拆分必须服务于可读性、可维护性、可测试性和复用性，不得为了增加文件或模块数量而机械拆分。对于逻辑简单、强相关且没有独立演进需求的代码，可以保持集中。
- 新增功能或重构现有代码时，应同步检查模块职责和依赖关系；若暂时无法合理解耦，应说明限制、风险及后续处理方案。

## 4. 游戏策划文档与资产留存

- 本仓库用于游戏开发，策划文档是项目的事实来源和最重要的核心资产之一，必须纳入版本控制并长期、完整保留。
- 每项工作都必须检查其对策划方案的影响。新增或修改玩法、系统、规则、数值、内容、交互、流程、资源、技术限制或项目范围时，必须在同一交付单元中同步新增或更新对应策划文档，不得只修改代码或资源。
- 策划文档必须描述当前有效方案，并保留重要方案的演进过程、变更原因、关键取舍和废弃内容。旧方案不得直接删除或无痕覆盖；不再生效的内容应明确标记状态，并迁移至历史记录或归档区域。
- 即使工作内容仅处于讨论、调研、原型或验证阶段，只要形成了可复用的结论、约束或决策，也必须写入策划文档，避免关键信息只存在于聊天、口头沟通、代码或 Agent 上下文中。
- 实现与策划文档发生冲突时，不得默认以代码现状覆盖策划方案。必须先识别差异，确认应调整实现还是修订方案，并保证最终代码、资源、测试与策划文档一致。
- 提交前必须检查策划文档是否同步、完整且可追溯；未完成必要的策划文档更新，视为任务未完成，不得以“修改完成”交付。

## 5. 外部美术与音频资产获取

- 当前项目不具备自行生产完整美术与音频资产的能力。开发需要图片、模型、动画、字体、特效、音乐、音效、配音或其他视听素材时，Agent 应主动从互联网检索并获取适用资产，不得因缺少现成素材而默认跳过相关实现。
- 只能使用许可证明确、来源可信且允许本项目预期用途的资产。优先选择 CC0、公有领域或明确允许商业使用及修改的资产；不得使用来源不明、授权条款缺失、仅限个人或非商业用途，或明显通过侵权方式转载的素材。
- 获取资产前必须核对并记录来源页面、作者、许可证名称、许可证链接、下载时间、允许的使用范围、署名要求和修改限制。资产文件应与对应授权证明或清单一并纳入版本控制，确保后续可审计和可替换。
- 使用需要署名、附带许可证文本或满足其他分发条件的资产时，必须同步维护游戏内外的署名与第三方声明。许可证彼此冲突或与项目发布方式不兼容时，不得引入。
- 网络资产必须经过安全与工程检查后再进入项目，包括文件格式、尺寸、质量、性能开销、恶意内容风险及风格一致性；不得执行来源不可信的脚本、插件、宏或可执行文件。
- 找不到授权和质量均合适的正式资产时，应使用可明确识别的临时占位资产，并在策划文档和任务记录中标注替换条件，不得将占位资产伪装成最终交付。
- 未经用户明确授权，不得购买付费资产、接受需要签署额外协议的条款，或注册、登录外部账号。遇到此类需求时，应先给出候选资产、成本、许可证和替代方案供用户决策。

## 6. 游戏表现力与反馈完整性

- 游戏功能设计与开发必须同时考虑表现层和反馈闭环，不得只实现逻辑正确但缺少视听反馈的“可运行版本”。每个面向玩家的功能都应评估动画与过渡、按钮及交互音效、环境与事件音效、视觉特效、Shader、镜头表现、界面反馈，以及目标平台支持时的震动或触觉反馈。
- 反馈设计必须服务于操作确认、状态变化、节奏、层级和情绪表达。关键操作、成功与失败、受击与命中、奖励、警告、冷却、不可用状态等应具有清晰且相互可区分的反馈，避免仅依赖单一视觉或声音通道传递必要信息。
- 动效、音效、特效和 Shader 应作为功能方案的一部分写入策划文档，明确触发条件、持续时间、优先级、可叠加规则、中断规则及预期效果；实现、资源和测试必须与文档同步。
- 表现层实现应保持模块化、数据驱动并与核心玩法逻辑解耦，使动画、音频、特效和 Shader 能够独立替换、调参、禁用和测试，不得将关键游戏判定依赖于某个表现效果是否播放完成。
- 所有表现效果必须考虑目标平台的性能预算和兼容性，包括 CPU、GPU、显存、Draw Call、粒子数量、Shader 复杂度、音频并发与包体大小。高成本效果应提供质量分级、降级或关闭方案，不得以明显影响帧率、输入响应或稳定性为代价换取表现。
- 必须考虑可访问性与用户控制：避免不可关闭的强烈屏闪、过度镜头晃动和突发高音量；必要时提供音量分类、动效简化、屏幕震动开关及其他适合项目的选项。
- 验收功能时必须同时验证玩法逻辑与表现反馈，包括不同状态切换、快速重复操作、中断、叠加、低性能配置及资源缺失场景。若当前阶段只能使用占位效果，必须明确记录缺失项、最终目标和替换条件。
